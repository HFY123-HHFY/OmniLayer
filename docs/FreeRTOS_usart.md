# FreeRTOS_usart

## 1. 这份文档是干什么的

这份文档说明当前工程中“RTOS 接管串口”的实现方式。

目标不是只让串口能收发，而是让你看懂：

- 为什么不能再用裸机那种全局变量 + 中断里直接处理业务的方式
- 当前串口收发的数据流是怎么走的
- Queue / Binary Semaphore / Mutex 在这里分别负责什么
- 以后如果要继续扩展命令解析、协议收发、上位机交互，应该从哪里下手

## 2. 设计目标

当前这套 RTOS 串口封装的目标是四个字：职责拆开。

具体来说：

- 中断只负责搬运字节和通知任务
- RX 后台任务负责组包
- TX 后台任务负责真正发送
- 业务任务只负责“要发什么”和“收到后怎么处理”

这样一来，串口链路就被拆成了清楚的几层，而不是所有逻辑都挤在中断和全局变量里。

## 3. 当前代码分层

### 3.1 `app/Control_Task/Control_Task.c`

这里保留 USART1 中断入口。

当前中断职责：

- RXNE 到来时，调用 `MyUsart_RtosRxIrqHandler(USART1)`
- TXE 到来时，调用底层 `usart_tx_irq_handler(USART1)`

也就是说，这一层主要是桥接，不负责高层业务。

### 3.2 `app/My_Usart/My_Usart.c`

这里是当前 RTOS 串口封装核心。

它负责：

- 创建队列
- 创建信号量
- 创建互斥锁
- 创建 TX / RX 后台任务
- 对外提供发送与接收接口
- 在中断和任务之间做数据衔接

### 3.3 `app/Control_Rtos/Control_Rtos.c`

这里是应用任务层。

它只关心：

- 初始化时启动串口 RTOS 封装
- 业务任务想发什么数据
- 收到完整一包数据后要怎么处理

## 4. 当前收发链路

## 4.1 接收链路

当前接收数据流如下：

```text
USART1 IRQ
   -> MyUsart_RtosRxIrqHandler()
   -> RX字节队列 s_rtosRxByteQueue
   -> RX二值信号量 s_rtosRxSem
   -> MyUsart_RtosRxTask()
   -> 组包
   -> RX行队列 s_rtosRxLineQueue
   -> MyUsart_RtosRecvLine()
   -> 上层业务任务
```

这条链路的含义是：

- 中断拿到的是“一个字节”
- 业务想处理的是“一整包文本”
- 所以中间必须有一个 RX 后台任务负责把字节拼成包

## 4.2 发送链路

当前发送数据流如下：

```text
业务任务
   -> MyUsart_RtosSendText() / MyUsart_RtosPrintf()
   -> TX消息队列 s_rtosTxMsgQueue
   -> MyUsart_RtosTxTask()
   -> 底层发送接口
   -> USART硬件发送
```

这样设计的价值是：

- 业务任务只负责“提交发送请求”
- 真正发送由一个统一的后台任务执行
- 避免多个任务并发抢占串口输出

## 5. Queue / Binary Semaphore / Mutex 在这里分别干什么

### 5.1 RX 字节队列

作用：传递“原始字节数据”。

- 生产者：USART 中断
- 消费者：RX 后台任务

为什么需要它：

- ISR 拿到字节时，不能在中断里做复杂组包
- 所以先把字节安全转交给任务

### 5.2 TX 消息队列

作用：传递“整段待发送文本”。

- 生产者：任意业务任务
- 消费者：TX 后台任务

为什么需要它：

- 业务任务不直接抢串口
- 统一由后台发送任务顺序处理

### 5.3 RX 行队列

作用：传递“已经组好的完整文本包”。

- 生产者：RX 后台任务
- 消费者：业务任务

为什么需要它：

- 上层任务不需要面对一个个字节
- 上层拿到的直接就是可处理的一条字符串

### 5.4 RX 二值信号量

作用：传递“有新数据到了”这个事件。

- 发送者：USART 中断
- 接收者：RX 后台任务

为什么不只靠队列：

- 队列解决的是数据搬运
- 信号量解决的是任务唤醒

这样 RX 任务可以阻塞等待，不需要空转轮询。

### 5.5 TX 互斥锁

作用：保护串口发送资源。

当前工程里虽然发送路径已经集中到 TX 后台任务，但保留互斥锁有两个现实价值：

- 如果以后扩展出更多共享发送资源，模型不需要推翻重来
- 从学习角度，你可以明确区分“消息排队”和“资源保护”是两件事

## 6. 当前接收成包策略

当前实现采用“双模式成包”：

- 如果收到 `\r` 或 `\n`，并且当前缓存里已经有数据，那么立即成包
- 如果一段时间没有继续收到新字节，也会按空闲超时自动成包

这样做的原因是为了兼容两种上位机使用习惯：

- 串口助手发送带回车换行
- 串口助手发送纯文本但不带换行

所以现在你无论发：

- `123\r\n`
- `123\n`
- `123`

都能被系统比较稳定地识别成一包。

## 7. 对外接口怎么理解

### 7.1 `MyUsart_RtosStart()`

作用：启动整套 RTOS 串口机制。

通常在初始化任务里调用一次。

它会做的事包括：

- 保存绑定的串口实例
- 创建各类队列
- 创建二值信号量
- 创建互斥锁
- 创建 TX / RX 后台任务

### 7.2 `MyUsart_RtosRxIrqHandler()`

作用：给串口 RX 中断使用。

它负责：

- 读取接收到的字节
- 把字节送到 RX 字节队列
- 给 RX 任务发送信号量通知

注意：

它不负责业务解析。

### 7.3 `MyUsart_RtosSendText()`

作用：发送一段字符串。

适合发送固定文本，例如：

- 状态信息
- 调试提示
- 命令回复

### 7.4 `MyUsart_RtosPrintf()`

作用：像 `printf` 一样格式化发送。

适合发送：

- 数字
- 浮点
- 组合字符串

例如业务任务可以反复调用它输出计数值、传感器值或状态值。

### 7.5 `MyUsart_RtosRecvLine()`

作用：从 RX 行队列里取出一条完整文本。

这说明上层任务拿到的不是零散字节，而是已经被 RX 后台任务组装好的结果。

## 8. 这套方案相比裸机升级了什么

如果对照裸机思路，可以这样理解：

裸机常见写法：

- 中断里接收字节
- 放全局缓冲区
- 改若干标志位
- 主循环再去轮询处理

当前 RTOS 写法：

- 中断只收字节并通知
- 后台任务负责组包
- 上层任务阻塞等待完整消息
- 发送统一走后台任务

升级点主要有四个：

1. 数据流清晰了。
2. 任务职责分开了。
3. 并发访问更安全了。
4. 后续扩展命令系统会容易很多。

## 9. 后面怎么扩展

在当前结构上，你后面可以继续做这些事情：

1. 在接收任务上层增加“命令解析任务”
2. 把 `Data: xxx` 回显改成真正的命令响应
3. 增加带帧头帧尾或 CRC 的协议层
4. 在串口发送侧继续封装日志等级或模块化输出

## 10. 你现在最应该掌握的理解

1. 串口中断只搬运字节，不做复杂业务。
2. RX 后台任务负责把字节变成消息。
3. TX 后台任务负责把业务请求变成串口发送动作。
4. Queue 传数据，Semaphore 传事件，Mutex 保护共享资源。
