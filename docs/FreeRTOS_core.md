# FreeRTOS_core

## 1. 这份文档是干什么的

这份文档面向从裸机切换到 FreeRTOS 的学习者，目标不是只讲 API，而是帮助你建立一套稳定的思维方式：

- 什么事情应该交给任务做。
- 什么事情必须留在中断里做。
- 为什么 RTOS 项目不能再用裸机那种“全局变量 + 中断里直接处理全部业务”的写法。
- 接下来如果要用 RTOS 管理定时器中断，应该按什么顺序学习。

## 2. 先建立一个最重要的分工原则

在 FreeRTOS 工程里，通常遵循下面这条原则：

- 中断负责“快收、快发、快通知”。
- 任务负责“计算、解析、打印、状态机、业务逻辑”。

也就是说，中断里只做最小动作，不做重活。

典型好写法：

- 串口中断里只把字节塞进队列，再唤醒任务。
- 定时器中断里只发事件、发计数、给信号量。
- 真正的数据处理放到任务里做。

这样做的原因有三个：

1. 中断越短，系统实时性越稳定。
2. 业务逻辑放在任务里，可读性和可维护性更好。
3. 任务上下文里可以安全使用更多 FreeRTOS 能力，而 ISR 环境限制很多。

## 3. FreeRTOS 最常用的几个核心对象

### 3.1 任务 Task

任务就是“被调度器管理的一段长期运行逻辑”。

你可以把它理解成裸机里的多个 `while(1)` 被操作系统拆开管理。

适合放进任务的内容：

- 周期业务
- 协议解析
- 数据融合
- 命令处理
- 后台发送
- 低优先级日志输出

不适合放进任务的内容：

- 必须在极短时间内响应的硬件边沿捕获第一现场

### 3.2 队列 Queue

队列适合传递“数据”。

它最重要的价值是把不同上下文解耦：

- ISR 把数据写入队列
- 任务从队列读取数据
- 任务 A 把消息发给任务 B

在当前工程串口封装里：

- RX 字节队列：中断写，RX 任务读
- TX 消息队列：业务任务写，TX 任务读
- RX 行队列：RX 任务写，业务任务读

所以你可以把队列理解成“线程安全的消息通道”。

### 3.3 二值信号量 Binary Semaphore

二值信号量适合传递“事件到了”的通知。

它本身不强调数据内容，而强调：

- 某件事发生了
- 某个任务可以继续运行了

典型场景：

- 中断收到数据后，通知接收任务
- DMA 完成后，通知处理任务
- 外部中断触发后，通知按键任务

当前串口方案里，RX 二值信号量的作用就是：

- USART IRQ 告诉 RX 任务：现在有新字节了
- RX 任务被唤醒后，再去队列里把字节取出来组包

### 3.4 互斥锁 Mutex

互斥锁适合保护“共享资源”。

它和二值信号量看起来都像“上锁/解锁”，但定位不同：

- 二值信号量更偏向事件通知
- 互斥锁更偏向资源保护

典型被互斥锁保护的对象：

- 串口发送通道
- I2C 总线
- SPI 总线
- 全局格式化输出缓冲区

如果多个任务都可能访问同一个共享硬件，通常需要 Mutex。

## 4. 为什么 RTOS 项目不能继续照搬裸机写法

裸机中经常这样写：

- 中断里直接接收数据
- 中断里直接解析协议
- 中断里直接设置多个全局标志
- 主循环再根据标志分支处理

这种方式在简单项目里能跑，但一旦任务增多就会出现问题：

- 全局变量越来越多
- 中断和主循环耦合严重
- 逻辑链路很难追踪
- 代码一大就容易竞争和时序错误

RTOS 的本质不是“加几个 API”，而是把系统拆成稳定的数据流与任务流。

## 5. 当前工程里建议保持的 RTOS 设计习惯

### 5.1 初始化只集中做一次

建议由一个初始化任务做这些事情：

- 板级资源初始化
- 驱动注册
- 外设初始化
- RTOS 后台服务启动
- 创建应用任务
- 自删

这样启动顺序清楚，也避免把初始化散落到多个地方。

### 5.2 中断只做桥接

例如：

- USART IRQ 只负责把字节送到队列并发信号
- TIM IRQ 只负责发通知或更新极小状态

不要在中断里直接写大段业务逻辑。

### 5.3 业务任务只关注业务

比如一个命令处理任务，它只关心：

- 有没有收到一包文本
- 文本内容是什么
- 应该回复什么

它不应该再关心硬件中断细节。

## 6. 接下来怎么学“RTOS 管理定时器中断”

这是你下一阶段最应该学的内容。

先说结论：

不要把“RTOS 管理定时器中断”理解成“定时器中断里直接跑 RTOS 业务”。

更准确的理解是：

- 定时器负责产生周期事件
- 中断负责把这个事件快速通知给任务
- 任务负责完成真正的周期业务处理

### 6.1 第一阶段：先学最小模式

先做一个最简单实验：

- TIM 定时 10ms 进入一次中断
- 中断里只做 `xSemaphoreGiveFromISR()` 或 `vTaskNotifyGiveFromISR()`
- 一个任务阻塞等待通知
- 每次收到通知后做一次 LED 翻转或计数打印

这一阶段要学会的不是功能，而是分工。

你要建立的直觉是：

- 定时器中断只是“拍一下肩膀”
- 真正干活的是任务

### 6.2 第二阶段：学周期基准任务

在 10ms 基准上，任务里再做软件分频：

- 10ms 执行一次按键扫描
- 50ms 执行一次姿态解算入口
- 100ms 执行一次状态上报
- 500ms 执行一次心跳灯

这比在多个定时器中断里分别塞业务更容易维护。

### 6.3 第三阶段：学“中断采样，任务处理”

适合传感器或编码器类场景：

- 定时器中断里只采一个最小原始量，或者只置位事件
- 任务里进行滤波、融合、PID、控制输出

这一步非常关键，因为很多控制工程会在这里犯错，把太多计算塞进 ISR。

### 6.4 第四阶段：学软件定时器和硬件定时器的区别

你需要区分两件事：

- 硬件定时器中断：由外设产生，实时性强，适合做节拍源、捕获触发、底层时基
- FreeRTOS 软件定时器：由内核定时器服务任务管理，适合较轻量的周期软件动作

建议初学顺序：

1. 先学硬件定时器 ISR 通知任务
2. 再学任务内基于 tick 的 `vTaskDelayUntil`
3. 最后再学软件定时器

### 6.5 第五阶段：开始关注中断优先级规则

一旦 ISR 要调用 `...FromISR()` 系列 API，你就必须关注中断优先级配置。

重点理解：

- 不是所有优先级的中断都能安全调用 FreeRTOS ISR API
- 需要满足 `configMAX_SYSCALL_INTERRUPT_PRIORITY` 约束
- 优先级配置错误时，常见现象是系统莫名死机、断言、任务不唤醒

所以你下一阶段不仅要学“怎么写”，还要学“哪些中断允许接 RTOS”。

## 7. 推荐学习顺序

建议你按这个顺序做实验：

1. 定时器中断 + 二值信号量 唤醒任务
2. 定时器中断 + 任务通知 唤醒任务
3. 任务里做多周期软件分频
4. 任务里接入传感器处理或控制算法
5. 对比硬件定时器方案与 `vTaskDelayUntil`
6. 再理解软件定时器

## 8. 你现在最应该记住的三句话

1. 中断负责最快速地捕获事件，不负责做完整业务。
2. 队列传数据，信号量传事件，互斥锁保资源。
3. RTOS 不是把裸机代码原封不动塞进任务里，而是重新整理数据流和职责边界。
