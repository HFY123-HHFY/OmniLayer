# FreeRTOS_core

## 1. 文档定位

这份文档不是当前工程某一个实验的说明书，而是一份面向学习者的 FreeRTOS 内核知识笔记。

它的目标有三个：

1. 帮助从裸机转到 RTOS 的开发者建立正确的思维方式。
2. 对 FreeRTOS 常见内核对象做系统性说明，方便后续查阅。
3. 让你在看工程代码时，知道“为什么这样设计”，而不是只会背几个 API。

如果你是第一次接触 RTOS，建议先看“任务”“优先级”“队列”“信号量”“中断规则”这几节。

## 2. 先理解什么是 RTOS

RTOS 是实时操作系统。这里的“实时”不是“跑得快”，而是“对事件的响应时间可预期”。

裸机常见模式是：

- 初始化一堆外设
- 进入一个大的 `while(1)`
- 中断里改标志位
- 主循环轮询这些标志位

RTOS 常见模式是：

- 初始化内核和硬件
- 创建多个任务
- 启动调度器
- 中断只做最小动作，任务负责业务处理

所以 RTOS 的核心价值不是“多开几个 while 循环”，而是：

- 把不同业务拆成独立执行单元
- 让不同任务按优先级和阻塞条件被调度
- 用内核对象来同步、通信和保护共享资源

## 3. FreeRTOS 的几个基础组成

一个最小的 FreeRTOS 系统通常由这些部分构成：

- 调度器 Scheduler
- 任务 Task
- 时基 Tick
- 中断服务与 `FromISR` API
- 通信对象（队列、信号量、通知等）
- 内存分配器（heap_1 ~ heap_5）

理解这些组成之后，你再看具体代码会容易很多。

## 4. 调度器 Scheduler 是什么

调度器的作用是决定“当前该让哪个任务运行”。

FreeRTOS 启动前，你创建任务只是把任务挂到内核管理结构里。真正开始调度，要等到调用：

`vTaskStartScheduler()`

调度器启动之后，CPU 会在这些情形下切换任务：

- 更高优先级任务就绪
- 当前任务主动阻塞
- 当前任务延时到期
- 中断唤醒了更高优先级任务
- 同优先级任务开启时间片轮转时发生 tick 切换

## 5. 任务 Task

### 5.1 任务是什么

任务就是一段被调度器管理的长期运行逻辑。

你可以把它理解成：

- 裸机里原本写在 `while(1)` 里的某一类职责
- 被拆出来交给内核调度

典型适合做成任务的内容：

- 控制循环
- 协议解析
- 串口命令处理
- 日志输出
- 显示刷新
- 后台数据搬运

不适合做成任务的内容：

- 必须在极短时间内完成的硬件第一现场捕获

### 5.2 任务的状态

FreeRTOS 任务常见状态有：

- Running：正在运行
- Ready：已经就绪，等待 CPU
- Blocked：因为等待事件/队列/延时而阻塞
- Suspended：被挂起，不参与调度
- Deleted：已删除

最关键的理解是：

一个设计良好的 RTOS 系统，任务大部分时间应该是 Blocked，而不是一直空转。

### 5.3 任务函数应该怎么写

典型任务结构：

```c
static void TaskExample(void *pvParameters)
{
	(void)pvParameters;

	for (;;)
	{
		/* 等待事件 */
		/* 处理业务 */
		/* 必要时进入阻塞 */
	}
}
```

核心原则：

- 任务应该有明确职责
- 任务内部尽量长期阻塞等待事件
- 不要让任务无意义地高速空跑

## 6. 优先级 Priority

### 6.1 优先级的含义

优先级决定“谁更应该先运行”。

在 FreeRTOS 里：

- 数值越大，优先级越高
- 高优先级任务会抢占低优先级任务

### 6.2 优先级不是越高越好

初学者最常犯的错误就是：

- 觉得重要的任务就一路把优先级拉高

这样会带来两个问题：

- 系统变得难以分析
- 低优先级任务可能长期得不到运行机会

更合理的原则是：

- 对实时性要求高的任务给更高优先级
- 对交互、日志、显示等低实时任务给较低优先级
- 初始化任务通常优先级可以高一点，但完成后应自删

### 6.3 当前工程里更推荐的任务分工方式

不是“一个功能一个任务”，而是“一个职责一个任务”。

例如：

- 系统初始化任务：只做一次启动流程
- 周期服务任务：统一承接 1ms 节拍，做软件分频
- 控制台任务：统一处理串口命令/回显

这比“点灯一个任务、打印一个任务、调试一个任务、计数一个任务”更清晰。

## 7. 栈 Stack 与任务栈大小

每个任务都有独立栈空间，用来保存：

- 局部变量
- 函数调用现场
- 上下文切换保存内容

任务栈过小会导致：

- 内存破坏
- 随机异常
- 奇怪死机

任务栈过大则会浪费宝贵 RAM。

所以任务栈大小不是“越大越安全”，而是“要够用且不过量”。

经验上：

- 字符串格式化、协议解析任务栈通常要大一些
- 简单周期任务可以小一些

## 8. 延时与周期控制

FreeRTOS 最常见的两种任务周期控制方式：

### 8.1 `vTaskDelay()`

适合“延时一段时间再继续”的场景。

特点：

- 简单
- 但周期会累计执行时间误差

### 8.2 `vTaskDelayUntil()`

适合“严格周期任务”。

特点：

- 周期更稳定
- 更适合固定采样周期的任务

### 8.3 硬件定时器通知任务

适合：

- 需要明确节拍源
- 中断和任务要协同工作
- 想把 1ms / 2ms / 10ms / 50ms 这些周期从一个硬件时基分发出去

这是当前工程学习定时器中断接 RTOS 的推荐方式。

## 9. 队列 Queue

### 9.1 队列解决什么问题

队列用于传输“数据”。

它的价值在于：

- 让 ISR 和任务之间安全传递数据
- 让任务和任务之间解耦

### 9.2 典型使用场景

- 串口接收字节流
- 任务之间发送消息结构体
- DMA / ADC 采样结果转交后台任务

### 9.3 你应该如何理解队列

不要只把它当作“一个 FIFO”。

在 RTOS 里，队列更重要的意义是：

- 数据通道
- 解耦边界
- 任务同步点

## 10. 信号量 Semaphore

信号量更适合传递“事件”或“资源数量”。

### 10.1 二值信号量 Binary Semaphore

适合：

- 中断通知任务
- 某个事件发生了，唤醒等待者

它更像是在说：

- “来活了”

### 10.2 计数信号量 Counting Semaphore

适合：

- 表示资源池数量
- 表示事件累计次数

比如多个同类资源可用，或者同一事件可能快速发生多次。

## 11. 互斥锁 Mutex

互斥锁用于保护共享资源。

典型保护对象：

- 串口发送通道
- I2C 总线
- SPI 总线
- 共享缓存

互斥锁和信号量的区别要记住：

- 信号量更偏向同步
- 互斥锁更偏向资源独占

FreeRTOS 中 Mutex 还带优先级继承，这对防止优先级反转很重要。

## 12. 优先级反转 Priority Inversion

这是 RTOS 很经典的概念。

假设：

- 高优先级任务等锁
- 低优先级任务拿着锁
- 中优先级任务一直抢 CPU

结果会导致高优先级任务反而被拖住，这就是优先级反转。

Mutex 的优先级继承机制就是为了解这个问题。

## 13. 事件组 Event Group

事件组适合管理“多个事件位”。

它适合这些场景：

- 等多个子模块准备完成
- 等多个状态位满足条件
- 管理复杂系统状态同步

如果你只是传单一事件，通常任务通知或二值信号量更轻量。

## 14. 任务通知 Task Notification

任务通知是 FreeRTOS 非常高效的一种同步机制。

它适合：

- 一个 ISR 唤醒一个特定任务
- 一个任务通知另一个固定任务

优点：

- 开销小
- 速度快
- 不需要额外创建独立对象

它的限制是：

- 更适合“一对一”通知
- 不适合复杂广播式同步

当前工程里 TIM3 节拍通知任务，就非常适合用任务通知。

## 15. 软件定时器 Software Timer

软件定时器是 FreeRTOS 内核自带的定时服务。

特点：

- 由内核定时器服务任务统一管理
- 使用方便
- 适合轻量周期动作

适合场景：

- 心跳灯
- 周期状态上报
- 软件超时处理

不太适合场景：

- 对实时性抖动极敏感的底层控制节拍

所以学习顺序建议是：

1. 先学硬件定时器 ISR 通知任务
2. 再学 `vTaskDelayUntil`
3. 最后再学软件定时器

## 16. 流缓冲区与消息缓冲区

FreeRTOS 还有两类很实用的通信对象：

- Stream Buffer
- Message Buffer

适合：

- 连续字节流搬运
- 按消息边界收发变长数据

如果你后面要学更复杂串口协议、DMA 接收、日志通道，这两类对象值得继续了解。

## 17. 中断与 `FromISR` API

### 17.1 中断里能做什么

推荐做法：

- 读取硬件寄存器
- 清中断标志
- 采集最小现场
- 投递队列/通知/信号量
- 必要时请求任务切换

不推荐做法：

- 大量 printf
- 大段协议解析
- 长时间计算
- 长时间 while 等待

### 17.2 为什么有 `FromISR` 版本

因为 ISR 环境和任务环境不同。

在 ISR 中调用普通任务 API 是不安全的，所以 FreeRTOS 提供了专门的 `...FromISR()` 版本。

### 17.3 `xHigherPriorityTaskWoken` 是什么

它用来告诉内核：

- 当前 ISR 是否唤醒了更高优先级任务

如果唤醒了，就可以在 ISR 末尾请求立即切换，让高优先级任务马上运行。

## 18. 中断优先级规则

这是 Cortex-M + FreeRTOS 中最容易踩坑的地方之一。

关键结论：

- 不是所有中断都可以调用 FreeRTOS API
- 只有优先级落在允许范围内的中断，才能安全调用 `FromISR()`

在工程里通常会涉及这些宏：

- `configPRIO_BITS`
- `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY`
- `configMAX_SYSCALL_INTERRUPT_PRIORITY`

如果配置错误，常见现象有：

- 任务不被唤醒
- 系统随机卡死
- 进入断言或 HardFault

所以当你写“TIM 中断 -> 通知任务”这类代码时，中断优先级一定要同步配置好。

## 19. 临界区与调度锁

### 19.1 临界区

临界区用于保护极短时间的共享数据操作。

常见 API：

- `taskENTER_CRITICAL()`
- `taskEXIT_CRITICAL()`

用途：

- 防止操作共享变量时被中断或切换打断

### 19.2 调度锁

调度锁只是不让任务切换，不等于关中断。

不要把它和临界区混为一谈。

## 20. 内存管理与堆

FreeRTOS 常见堆实现有：

- `heap_1`
- `heap_2`
- `heap_3`
- `heap_4`
- `heap_5`

当前工程使用的是 `heap_4`。

它的特点是：

- 支持分配和释放
- 带内存块合并
- 比较适合一般嵌入式项目

但你仍然要记住：

- 动态分配不是没有代价
- 任务、队列、信号量开多了都会吃 RAM

## 21. Hook 与调试辅助

FreeRTOS 还能提供一些调试或监控辅助机制，例如：

- 栈溢出检测
- malloc 失败钩子
- 空闲任务钩子
- Tick 钩子

在项目进入稳定阶段时，建议适当打开这些诊断能力。

## 22. RTOS 设计习惯建议

### 22.1 初始化任务只做一次

典型职责：

- 注册板级资源
- 初始化硬件
- 创建常驻任务
- 自删

### 22.2 中断只做桥接

例如：

- USART IRQ 只收字节并通知任务
- TIM IRQ 只发节拍通知

### 22.3 任务按职责分层

更推荐的任务划分方式是：

- 周期服务任务
- 控制任务
- 通信任务
- 显示任务
- 日志任务

而不是：

- 点灯一个任务
- 打印一个任务
- 每个小实验单独一个任务

## 23. 对裸机开发者最重要的思维转换

你需要从这几种裸机习惯里走出来：

- 中断里直接做完全部业务
- 主循环疯狂轮询标志位
- 大量全局变量耦合模块

换成 RTOS 思维以后，应当变成：

- 中断收现场，任务做业务
- 队列传数据，通知/信号量传事件
- Mutex 保护共享资源
- 任务大部分时间阻塞等待，不空转

## 24. 如何学习“RTOS 管理定时器中断”

建议学习顺序：

1. 定时器中断只通知一个任务
2. 任务里做软件分频
3. 任务里托管 2ms / 10ms / 50ms / 1s 周期逻辑
4. 再把传感器处理、控制算法逐步接进来
5. 最后对比硬件定时器、`vTaskDelayUntil`、软件定时器三种方案

## 25. 最后记住这几句话

1. RTOS 的重点不是 API，而是职责边界。
2. 中断越短，系统越稳。
3. 任务不是越多越好，而是职责越清楚越好。
4. 优先级不是越高越好，而是越合理越好。
5. 队列传数据，信号量传事件，互斥锁保资源，任务通知做轻量一对一同步。

典型使用方式：

- ISR 写队列，任务读队列
- 任务 A 写队列，任务 B 读队列

队列的价值在于把生产者和消费者解耦。

### 9.1 适合使用队列的场景

- 串口逐字节接收
- 任务间命令消息传递
- 采样值从采集任务送到处理任务

### 9.2 队列不适合替代一切

如果你只是想表达“事件发生了”，而不是传递一段数据，那么任务通知或信号量通常更轻。

## 10. 信号量 Semaphore

信号量更适合表达“同步关系”或“事件到了”。

### 10.1 二值信号量 Binary Semaphore

适合：

- ISR 通知任务有新事件
- DMA 完成通知处理任务
- 外部中断通知按键任务

它强调的是“通知”，不是“数据内容”。

### 10.2 计数信号量 Counting Semaphore

适合：

- 资源池数量管理
- 统计事件次数
- 多个事件积压时不丢次数

## 11. 互斥锁 Mutex

互斥锁适合保护“共享资源”。

例如：

- 串口发送通道
- I2C 总线
- SPI 总线
- 公共格式化缓冲区

### 11.1 Mutex 和二值信号量的区别

- 二值信号量更偏向同步事件
- Mutex 更偏向资源保护

### 11.2 优先级继承

Mutex 的一个重要价值是支持优先级继承，能缓解优先级反转问题。

## 12. 事件组 Event Group

事件组适合表示“多个条件位”。

典型场景：

- 等待多个模块初始化完成
- 等待多个传感器就绪
- 某任务需要同时关注多个系统状态位

如果你要表达的是“若干布尔条件的组合状态”，事件组往往比多个单独 flag 更清晰。

## 13. 任务通知 Task Notification

任务通知是 FreeRTOS 里非常高效的一种同步机制。

它的特点：

- 不需要额外创建单独对象
- 直接挂在任务控制块上
- 开销通常比队列和信号量更小

### 13.1 适合任务通知的场景

- 一个 ISR 只唤醒一个任务
- 一个硬件事件只对应一个处理任务
- 周期节拍只需要通知某一个任务

你当前工程的 TIM3 1ms 节拍通知，就是一个典型使用场景。

## 14. 软件定时器 Software Timer

FreeRTOS 软件定时器不是硬件定时器。

它由内核里的 Timer Service Task 统一管理。

适合：

- 轻量周期动作
- 非强实时的超时管理
- 协议超时、看门狗式软件状态检查

不适合直接替代：

- 高精度控制周期
- 强实时采样触发

## 15. Stream Buffer / Message Buffer

这两个对象常被初学者忽略，但也很有用。

### 15.1 Stream Buffer

适合连续字节流：

- 串口流式数据
- DMA 接收到的一串数据

### 15.2 Message Buffer

适合一条一条的变长消息：

- 文本包
- 协议帧
- 可变长度命令消息

如果以后你的串口协议从“按字节组包”升级成“消息帧处理”，这两个对象也值得对比学习。

## 16. 临界区 Critical Section

临界区用于保护很短的共享代码段，避免被中断或任务切换打断。

常见接口：

- `taskENTER_CRITICAL()`
- `taskEXIT_CRITICAL()`

使用原则：

- 临界区必须短
- 不要在临界区里做耗时动作
- 能用 Mutex 的共享资源，不要滥用临界区替代

## 17. 延时与周期执行

常见延时方式：

- `vTaskDelay()`：延时若干 tick
- `vTaskDelayUntil()`：按固定周期运行，更适合周期任务

### 17.1 两者区别

- `vTaskDelay()` 更像“从现在起睡多久”
- `vTaskDelayUntil()` 更像“严格按节拍点执行”

如果以后你想对比“硬件定时器通知任务”和“任务自己按 tick 周期运行”，就重点看这里。

## 18. FreeRTOS 中断规则

只要 ISR 要调用 `...FromISR()` 系列 API，你就必须关心中断优先级。

关键点：

- 不是所有中断优先级都能安全调用 FreeRTOS ISR API
- 中断优先级必须落在 `configMAX_SYSCALL_INTERRUPT_PRIORITY` 允许范围内
- 默认最高优先级中断通常不能直接调用内核 API

### 18.1 典型后果

如果优先级配置错了，常见现象包括：

- 任务不被唤醒
- 系统随机死机
- 某些中断一来系统就异常

### 18.2 一条实用经验

只要某个硬件中断里要用队列、信号量、任务通知这些 FromISR API，就应该显式设置它的 NVIC 优先级，而不要依赖默认值。

## 19. FreeRTOS 常见钩子与调试手段

常见可选功能：

- 栈溢出检测
- malloc 失败钩子
- 空闲钩子
- tick 钩子

调试时建议关注：

- 任务是否真的阻塞而不是空转
- 栈是否够用
- 中断是否只做最小工作
- 是否误用了高优先级中断调用内核 API

## 20. 什么时候该新建任务

你后面写代码时，可以按下面这套判断：

### 应该新建任务的情况

- 它有独立的阻塞等待对象
- 它有独立的周期调度需求
- 它和其它业务明显属于不同职责域
- 它需要长期常驻并被单独调度

### 不应该新建任务的情况

- 它只是另一个任务中的一个步骤
- 它没有自己的阻塞点
- 它没有自己的时间需求
- 它只是一个普通功能函数

## 21. 当前工程推荐的任务思路

在当前工程里，更推荐这样理解任务：

- `TaskSystemInit`：一次性启动任务
- `TaskPeriodicService`：统一管理 1ms/2ms/50ms/1s 等周期业务
- `TaskConsoleService`：统一处理串口控制台收发

这比“2ms 一个任务、50ms 一个任务、打印一个任务、点灯一个任务”更适合学习，也更贴近工程实践。

## 22. 推荐学习顺序

建议你按下面顺序补知识：

1. 任务、状态、优先级、阻塞
2. 栈、堆、动态分配
3. 队列、二值信号量、互斥锁
4. 任务通知、事件组
5. 硬件中断与 `FromISR` API
6. `vTaskDelayUntil()` 与软件定时器
7. 优先级反转、实时性、系统设计取舍

## 23. 最后记住这几句话

1. RTOS 不是把裸机代码搬进任务，而是重新整理职责边界。
2. 任务是“调度单位”，函数是“逻辑单位”，两者不是一回事。
3. 队列传数据，信号量传事件，互斥锁保资源，任务通知做单点唤醒很高效。
4. 中断必须短，复杂逻辑尽量下沉到任务。
5. 不是功能越多任务越多，而是职责越独立、调度需求越不同，才越值得拆任务。
