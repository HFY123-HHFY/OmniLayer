# FreeRTOS_timer

## 1. 文档目标

本文档用于说明本工程如何把“定时器中断驱动逻辑”迁移为“RTOS 任务驱动逻辑”，并给出可复用的设计原则。

关键词：

- TIM 中断
- Task Notification
- 周期任务
- 软件分频
- 低优先级遥测

## 2. 设计结论（先看这个）

在 FreeRTOS 分支中，定时器中断不再承担复杂业务计算，而只负责节拍通知。

推荐模式：

1. TIM IRQ：清中断标志 + 发送通知
2. 周期任务：消费通知，做软件分频与控制入口
3. 遥测任务：低优先级打印，避免影响控制周期

这样做的核心目的是“稳定周期任务时序”。

## 3. 为什么不在定时器中断里直接做业务

中断里做重活会带来这些问题：

- ISR 执行时间不可控
- 高优先级中断长期占用 CPU
- 任务调度抖动变大
- 系统行为难以调试和维护

因此本工程遵循：

- ISR 只做最小动作
- 复杂逻辑放任务

## 4. 当前工程的定时器接管链路

以 TIM3 为例，当前链路是：

```text
TIM3 IRQ (1ms)
  -> xTaskNotifyGiveFromISR(periodicTask)
  -> TaskPeriodicService
  -> 软件分频（2ms/50ms/1s）
  -> 通知 TaskTelemetryService 打印
```

其中：

- TIM3 IRQ 只负责“发节拍”
- TaskPeriodicService 负责“做节拍分发”
- TaskTelemetryService 负责“做串口调试打印”

## 5. 为什么周期任务不直接打印

周期任务里直接 `MyUsart_RtosPrintf` 短期看能工作，但存在风险：

- 串口输出耗时可变
- 周期任务执行时长被拉长
- 2ms/10ms 等实时逻辑抖动增大

本工程采用“通知低优先级遥测任务打印”的方式，是为了让周期任务尽量短、尽量稳。

## 6. 通知机制说明

本工程定时器接管使用 Task Notification：

- 中断到周期任务：`xTaskNotifyGiveFromISR` + `ulTaskNotifyTake`
- 周期任务到遥测任务：`xTaskNotifyGive` + `ulTaskNotifyTake`

优点：

- 开销低
- 一对一唤醒简单直接
- 适合节拍触发型链路

## 7. 周期任务中的软件分频示例

常见软件分频策略：

- 每次收到 1ms 通知，计数器加 1
- 到 2 时执行 2ms 逻辑
- 到 50 时执行 50ms 逻辑
- 到 1000 时执行 1s 逻辑

这比“多个定时器中断分别塞逻辑”更容易维护和扩展。

## 8. 动态调试周期（rate 命令）

控制台任务支持 `rate <ms>` 动态修改遥测输出周期。

示例：

- `rate 100`：把调试打印改为 100ms
- `rate?`：查询当前周期

通过该机制可以在线调试，而不用频繁改代码重编译。

## 9. F103/F407 差异体现在哪里

RTOS 接管逻辑本身保持一致，但芯片差异通过以下层体现：

1. Core 层定时器实现不同（寄存器/NVIC 初始化）
2. FreeRTOSConfig 不同（CPU 主频、堆大小等）
3. CMake 预设与 OpenOCD 配置不同

也就是说：

- 架构逻辑统一
- 底层适配分芯片

## 10. 常见问题排查

### 问题1：看起来没进定时器链路

检查：

- 是否调用了 API_TIM_Init
- 中断是否注册到正确 IRQ
- 任务句柄是否成功注册

### 问题2：F103 编译了但板子没反应

检查：

- 构建预设是否是 Debug-F103
- 烧录任务是否选了 F103 目标
- 实际连接板卡是否真的是 F103

### 问题3：打印一开，周期变乱

检查：

- 是否在周期任务里直接大量打印
- 是否应改为“周期任务通知 -> 低优先级遥测任务打印”

## 11. 推荐实践

1. ISR 永远短小，只发通知。
2. 周期任务只管节拍和业务入口，不做重打印。
3. 串口打印统一低优先级任务承接。
4. 调试周期支持命令动态调整，减少重编译次数。
5. 切 MCU 时优先使用 Select MCU 任务，避免目标错配。
